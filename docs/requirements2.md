# Requirements Document: Сервис доставки еды

1. Функциональные требования (ФТ) 
    - ФТ-001: Пользователь может искать рестораны по названию, типу кухни, рейтингу, расстоянию и времени доставки
    - ФТ-002: Пользователь может открыть меню ресторана и посмотреть категории блюд, фото блюда, описание и цену, наличие, (пищевую ценность???)
    - ФТ-003: Пользователь может добавлять блюда в корзину (из разных ресторанов???), выбрать способ доставки, указать адрес доставки, подтвердить заказ
    - ФТ-004: Пользователь может оформить заказ и оплатить его онлайн
    - ФТ-005: Пользователь может отслеживать статус заказа в реальном времени после оформления: принят/готовится/передан курьеру/в пути/доставлен
    - ФТ-006: Пользователь может получать push-уведомления при изменении статуса заказа
    - ФТ-007: Ресторан может принимать/отклонять заказы, видеть список активных заказов, менять статус заказа на принят/готовится/передан курьеру
    - ФТ-008: Пользователь может оставить отзыв о ресторане/блюде (только после заказа в этом ресторане???)

3. Расчет нагрузки (Capacity Estimation)

    3.1 Пользователи

    MAU: 20'000'000

    DAU: 3'000'000 = MAU * 0.15 , обоснование: типичный DAU/MAU для food services ~10-25% т.к. люди заказывают еду не каждый день, а скорее по необходимости

    Пиковый одновременный онлайн: 150'000 = DAU * 0.05, обоснование: обычно у похожих сервисов пиковый онлайн ~3-8% т.к. сессии короткие, а пользователи заходят переодически

    3.2 RPS

        Соотношение чтение/запись: 85/15 , обоснование: пользователи в основном просматривают рестораны и меню, а не оформляют заказы

        Средний RPS = DAU * Req/user / 86400
            = 3'000'000 * 50 / 86400
            = 1750 RPS [R/W: 1500/250]

        Peak_hours: 12-14 (обеденный пик) и 18-21 (вечерний пик) [5ч]
        Peak_coef: x3 (типичное значение)
        Peak RPS    = (DAU × Req/user × Peak_coef) / (Peak_hours × 3600)
            = (3'000'000 × 50 × 3) / (5 × 3600)
            = 25'000 RPS [R/W: 21000/4000]


    3.3 Бюджет задержки (Latency Budget)

        Endpoint: "GET /api/restaurants" SLO: p99 < 300ms

   
    | Компонент | Задержка (мс) | Комментарий |
    | --------- | ------------- | ----------- |
    | API Gateway | 10 | сетевой hop, маршрутизация |
    | Auth | 5 | легкая аутентификация |
    | Restaurant Service | 50 | выборка данных из бд/кэша |
    | DB query | 100 | поиск в бд |
    | Caching layer | -20 | cache hit ускорит поиск в бд|
    | Serialization + Response | 15 | формирование и отправка response |
    | ИТОГО | 160 |
    | Буфер (30%) | 48 |
    | С БУФЕРОМ | 208 |
    | Остаток от SLO | 92 | (можно добавить рекомендации) |

        Узкое место (Bottleneck): DB query при промахе по кэшу - можно принудительно кэшировать популярные рестораны, read replicas, lazy loading
    

        Endpoint: "POST /api/order" SLO: p99 < 1300ms
   
    | Компонент | Задержка (мс) | Комментарий |
    | --------- | ------------- | ----------- |
    | API Gateway | 10 | сетевой hop, маршрутизация |
    | Auth | 5 | легкая аутентификация |
    | Order Service | 100 | Валидация корзины, обработка запроса |
    | DB write | 200 | Транзакционная запись + WAL + репликация |
    | Paywment Gateway | 600 | внешняя обработка платежа |
    | Event queue | 50 | добавляем оформленный заказ в очередь событий |
    | Serialization + Response | 15 | формирование и отправка response |
    | ИТОГО | 980 |
    | Буфер (30%) | 294 |
    | С БУФЕРОМ | 1274 |
    | Остаток от SLO | 26 ||

        Узкое место (Bottleneck): Paywment Gateway - асинхронаая обработка платежей с идемпотентностью + отпускаем пользователя с экрана сразу и потом присылаем push, запись заказа до финальной оплаты, локальное сохранение данных о карте

    3.4 Хранилище (Storage)

        Размер заказа:
            - Order record: ~2 Кб
            - Items: ~1 Кб
            - Status history: ~1 Кб
            - Total: 4 Кб
        
        Размер фото: ~1 Mб

        Storage = записей/день × размер_записи × дней × коэффициент_репликации

        Заказы:
        3'000'000 DAU * 1.5 заказа/день * 4 KB/заказ = 17 GB/день
        × 365 = ~6 TB/год
        × 3 (репликация) = ~18 TB/год

        Фото меню:
        50K ресторанов × 25 блюд × 1 MB/фото = 1.2 TB (единовременно)
        Обновление ~5%/мес = 62 GB/мес

       
    | Тип данных | Объем через 1 год | Объем через 3 года | Комментарий |
    | ---------- | ----------------- | ------------------ | ----------- |
    | Заказы | 20 TB | 60 TB | 4.5M заказов в день + x3 репликация |
    | Фото меню | 2 TB | 3.5 TB | Начальный объем 1.2 TB +5%/мес |
    | Каталог ресторанов и меню | 50 GB | 100 Gb| текстовые данные, цены  (по ~1Gb на ресторан) |
    | Логи | ~1 TB | ~3 TB ||

    3.5 Bandwidth
        Формулы:

        Входящий  = Peak_RPS × avg_request_size
        Исходящий = Peak_RPS × avg_response_size

        Avg:
            - Входящий:  1750 RPS × 2 KB  = 3 MB/s = 24 Mbps
            - Исходящий: 1750 RPS × 50 KB = 250 MB/s = 700 Mbps
        Peak:
            - Входящий:  25K RPS × 2 KB  = 10 MB/s = 400 Mbps
            - Исходящий: 25K RPS × 50 KB = 250 MB/s = 10 Gbps

4. Допущения, ограничения и открытые вопросы
 - Допущения 
    1. Каталог ресторанов ~50K
    2. Среднее меню ресторана 25 блюд
 - Ограничения 
    1. Потенциально высокая сетевая задержка
    2. Многие иностранные сервисы заблокированы
    3. Инфраструктура должна быть экономичной
    4. Необходим auto-scaling у облака
 - Открытые вопросы 
    1. Сколько ресторанов планируется в системе?
    2. Нужна ли поддержка маршрутов для курьеров?
    3. Какие платёжные системы используются?
    4. Нужна ли персонализация поиска?
    5. Как долго надо хранить "старые" заказы?
 - Выявленные противоречия [Какие требования заказчика конфликтуют друг с другом и как вы предлагаете их разрешить.]
    1. “Бюджет ограничен” vs “Высокая доступность” → Снижение нагрузки через кэширование, использование по большей части read replicas, auto-scaling только в часы пика
    2. “Ничего не должно теряться” vs “очень быстро” → идемпотентые запросы к API + async обработка
