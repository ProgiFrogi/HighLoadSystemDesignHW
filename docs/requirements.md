# Requirements Document: Сервис доставки еды

## ТЗ и вводные

- Каталог ресторанов и меню
- Оформление и оплата заказов
- Отслеживание статуса доставки в реальном времени
- Уведомления (push/sms)

Вводные от заказчика:

«Система должна выдерживать обеденный пик. Оплата — самое важное, ничего не должно теряться. Но скорость оформления тоже критична — клиенты уходят, если ждут больше пары секунд.»

Ожидаемая аудитория: 5–20 млн пользователей в месяц
Регион: Россия + СНГ
Инфраструктура: облако, бюджет ограничен
«Система должна работать быстро и не падать»

## Задание

### 1. Функциональные требования(ФТ)

- ФТ-001: Пользователь может найти ресторан по названию, кухне или расстоянию, рейтингу, времени доставки
- ФТ-002: Пользователь может получить данные о блюде из меню
- ФТ-003: Пользователь может добавлять блюда в корзину, выбрать способ доставки, указать адрес, оформить заказ
- ФТ-004: Пользователь может оформить заказ онлайн и оплатить его онлайн/при получении
- ФТ-005: Пользователь может отслеживать статус доставки и расстояние курьера до адреса доставки
- ФТ-006: Пользователь может получить уведомление о статусе заказа
- ФТ-007: Ресторан может получить данные о заказе пользователя: что в нем, кем он доставляется/доставлялся, отзывы пользователя, изменять статус заказа
- ФТ-008: Заказчик может модерировать заявки и рестораны: списываться с их представителями, удалять/приостанавливать/добавлять ресторан на площадке

## Оценка нагрузки (Capacity Estimation)

### 3.1 Пользователи

`MAU = 5–20 млн пользователей в месяц`

Предположим, что MAU = 20 млн пользователей в месяц.

Предположим, что пользователи заказывают 1-2 раза в неделю(так мы учитываем(предположительно) и доставку продуктов, и доставку из ресторанов), это 4-8 заказов за месяц. Возьмем 6 как количество заказов в месяц для пользователя.
Тогда наш пользователь активен в 6 днях из 30, значит

```
Stickiness = 6/30= 20% (что взято с хорошим запасом).
```

Тогда

```
DAU = 20 * 0.2 = 4 (млн. пользователей)
```

Для пиков:
Предположим, что пик достигается в обед (12:00-14:00) и на ужин (18:00-20:00).
И пусть 40% заказов - обед, 30% - ужин, 30% - распределено.
Тогда обеденный пик:

```
Peak = DAU*0.4 = 4 * 0.4 = 1.6 млн. пользователей
```

Ужинный пик:

```
Peak = DAU*0.3 = 4 * 0.3 = 1.2 млн. пользователей
```

Пиковый одновременный онлайн: 1.6 млн. пользователей (возьмем самый большой пик)

### 3.2 RPS

Предположим стандартный пользовательский сценарий
с учетом операции на Read(R) и Write(W):
(по порядку)

- 1-2R (Просмотр ресторанов)
- 2-5R (Просмотр меню)
- W (Добавление заказа в корзину)
- R (переход в корзину)
- W (подтверждение заказа)
- API+W (оплата заказа)
- W+R (получение рестораном заказа)
- W+R (уведомления (заказ принят и готовится/заказ передан курьеру/заказ доставлен на адрес/заказ получен))
- W (отправка отзыва пользователем (например, в звездах))

При  этом нужно учитывать:

- Большая часть пользователей будет в основном смотреть, а не делать заказ.

Предположим конверсия 20%. Тогда на 1 пройденный путь с 8R, 5W будет 4 пути с 12R(чисто читающий пользователь).
Получаем 32R/5W или 85/15

```Соотношение чтение/запись: 85/15```
Теперь, предположив конверсию в 20% получим:

```
Req/user = (0.8 x 12) + (0.2 x 13) = 12.2 (возьмем 13 c запасом)
```

Тогда

```
AvgRPS = DAU x Req/user / 86400 = 4млн * 13 / 86400 = 601 RPS ~ 600 PRS
```

Для расчета `PeakCoef`, если считаем, что в эти 2 часа приходят 40%:

```
UserInPeakHour/Hour = DAU * 0.4 / 2 = 4млн * 0.4 / 2 = 800 000 user/hour
```

```
AvgUser = DAU / 24 = 166 666 user/hour
```

```
PeakCoef = UserInPeakHour/Hour / AvgUser = 800000 / 166666 = ~ 4.8
```

```
PeakRPS =  (DAU * Req/user * Peak_coef) / (Peak_hours * 3600)
= 4000000 * 13 * 4.8 / (2 * 3600) = 34 666 RPS ~ 35000 RPS
```

### 3.4 Хранилище (Storage)

```
Storage = записи/день x размер_записи x дней x коэффициент репликации
```

Заказы:

- DAU=4 млн., 20% пользователей делает заказы => 1млн заказов/день
- Размер записи ~5КВ (ID, ID пользователя, состав заказа, суммы, статусы, время)
- Объем/день: `800000 * 5KB = 4 GB/день`
- Объем/год: `365 * 4GB = ~1460GB`
- С репликацией (x3): `~4.4TB/год`
- Рост:
  - Пусть на 10% больше DAU/год
  - Тогда за 2й год: `1.1 * 4.4 = 4.8TB`
  - За 3й: `1.1 * 4.8 = 5.3 TB`

Профили пользователей:

- Записей ~20 млн. MAU + 8 млн. неактивных (с запасом)
- Размер записи: `~2KB` (имя, телефон, email, пароль)
- Объем: `28 x 2KB = ~56GB`
- Адреса доставки: пусть 2 адреса по 0.5KB: `~28GB`
- Итого: `84GB`
- С репликацией (x3): `252GB`
- Рост:
  - Предположим, что на 10% больше пользователей в год:
  - 2й год: `277GB`
  - 3й год: `305GB`

Меню ресторанов:

- Количество ресторанов: `20k`
- Блюд на ресторан (в среднем): `50`
- Размер блюда: `~0.5KB` (это название+цена+категория+описание - примерно 500 символов)
- Размер фото: `~100KB`(сжатые)
- Объем на ресторан: `50 * (0.5 + 100) = ~5MB`
- Итого: `5MB x 20k = ~98GB`
- Репликация (x3) `~294GB`
- Обновления(новых блюд в ресторанах): `2.5 блюд/месяц = 15GB/мес` => `+180GB`
- Итого `474GB`
- Рост:
  - Пусть на 10% больше ресторанов каждый год:
  - 2й год: `474GB * 1.1 = 522GB`
  - 3й год: `575GB`

Отзывы:

- Записи/день: пусть в 2% заказов оставляют отзывы: `16000 отзыв/день`
- Размер отзыва(фото опционально, далее учтем): `~1KB`
- Объем/день (пусть фото в 5% отзывов, фото в среднем 500KB): `16000 * (0.95 * 1KB + 0.05 * 500KB) = ~0.4GB`
- Объем/год: `~145GB/год`
- Репликация (x2): `~290GB/год`
  - 2й год: `~319GB/год`
  - 3й год: `~351GB/год`

### 3.5 Bandwidth
    Формулы:

    Входящий  = Peak_RPS * avg_request_size
    Исходящий = Peak_RPS * avg_response_size

    Avg:
        - Входящий:  600 RPS * 2 KB  = 1.2 MB/s = 9.6 Mbps
        - Исходящий: 600 RPS * 50 KB = 29 MB/s = 232 Mbps
    Peak:
        - Входящий: 35K RPS * 2 KB = 68 MB/s = 544 Mbps
        - Исходящий: 35K RPS * 50 KB = 1.67GB = 13.4 Gbps
