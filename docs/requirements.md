# 1. Функциональные требования (ФТ)

**ФТ-001: Поиск ресторанов**

Пользователь может искать рестораны:
- по названию
- по кухне
- по рейтингу
- по расстоянию
- по фильтрам (доставка, цена, акции)

---

**ФТ-002: Просмотр меню**

Пользователь может:
- открыть страницу ресторана
- просматривать категории меню
- смотреть описание блюд, цену, фото

---

**ФТ-003: Оформление заказа**

Пользователь может:
- добавить блюда в корзину
- указать адрес доставки
- выбрать способ оплаты
- подтвердить заказ

---

**ФТ-004: Онлайн-оплата**

Пользователь может оплатить заказ:
- банковской картой
- через внешний платежный сервис

Оплата является **критичной операцией — потеря платежа недопустима**.

---

**ФТ-005: Отслеживание статуса заказа**

Пользователь может видеть статус заказа:

- принят
- готовится
- передан курьеру
- в пути
- доставлен

Статус обновляется **в реальном времени**.

---

**ФТ-006: Уведомления**

Пользователь получает уведомления:

- подтверждение заказа
- статус доставки
- завершение заказа

Каналы:

- push
- SMS

---

**ФТ-007: Управление заказами**

Пользователь может:

- просматривать историю заказов
- повторять предыдущие заказы
- оставлять отзывы

---

# 3. Оценка нагрузки (Capacity Estimation)

## 3.1 Пользователи

По вводным заказчика ожидаемая аудитория составляет **5--20 млн MAU**.\
Для проектирования берём верхнюю границу.

MAU = 20 000 000

Типичное соотношение:

DAU ≈ 20% MAU

DAU = 4 000 000 пользователей

Во время обеденного и вечернего пиков обычно онлайн находится около
**10% DAU**.

Peak concurrent users:

4 000 000 × 10% = **400 000 пользователей**

------------------------------------------------------------------------

## 3.2 Поведение пользователя

Предположим типичную пользовательскую сессию:

  Действие                 Количество
  ------------------------ ------------
  Поиск ресторанов         3
  Просмотр меню            5
  Просмотр карточек блюд   4
  Оформление заказа        0.2

Итого:

≈ **12 запросов на пользователя**

------------------------------------------------------------------------

## 3.3 Общее количество запросов

Requests per day = DAU × requests_per_user

4 000 000 × 12 = 48 000 000 запросов / день

------------------------------------------------------------------------

## 3.4 Средний RPS

Используем правило:

1 день ≈ 100 000 секунд

RPS = 48 000 000 / 100 000

≈ 480 RPS

------------------------------------------------------------------------

## 3.5 Пиковая нагрузка

Пиковая нагрузка может быть **в 4--5 раз выше средней** (обеденный пик).

Peak RPS ≈ 480 × 5

Peak RPS ≈ 2400

------------------------------------------------------------------------

## 3.6 RPS по операциям

  Операция            Доля   Avg RPS   Peak RPS
  ------------------- ------ --------- ----------
  Поиск ресторанов    35%    170       850
  Просмотр меню       30%    140       700
  Просмотр блюд       20%    95        480
  Оформление заказа   10%    48        240
  Оплата              5%     24        120

------------------------------------------------------------------------

## 3.7 Latency Budget (оформление заказа)

Целевое SLO:

p99 \< **700 ms**

Разложим путь запроса.

Client → API Gateway → Auth Service → Order Service → Database → Payment
Provider → Notification Service

| Компонент | Latency |
|---|---|
| API Gateway | 10 ms |
| Auth Service | 15 ms |
| Order Service | 30 ms |
| Database write | 5 ms |
| Payment provider API | 300 ms |
| Notification Service | 20 ms |
| Network overhead | 50 ms |

Итого:

≈ **430 ms**

Запас:

700 − 430 = **270 ms**

Основной bottleneck --- **внешний платежный сервис**.

------------------------------------------------------------------------

## 3.8 Хранилище

### Заказы

Структура заказа:

-   order_id
-   user_id
-   restaurant_id
-   items
-   price
-   status
-   timestamps

Размер записи:

≈ **1 KB**

Количество заказов:

Предположим **5% DAU делают заказ ежедневно**.

Orders per day = 4M × 5%

= **200 000 заказов**

В год:

200 000 × 365 ≈ **73 000 000 заказов**

Размер:

73M × 1 KB ≈ **73 GB**

С индексами и метаданными:

≈ **200 GB / год**

Через 3 года:

≈ **600 GB**

------------------------------------------------------------------------

### Каталог ресторанов

Предположим:

100 000 ресторанов

Среднее меню:

50 блюд на ресторан

100 000 × 50 = **5 000 000 блюд**

Размер записи блюда:

≈ **500 B**

5M × 500 B = **2.5 GB**

С индексами:

≈ **5 GB**

------------------------------------------------------------------------

## 3.9 Bandwidth

Средний размер ответа API:

≈ **50 KB**

Peak RPS:

2400

Исходящий трафик:

2400 × 50 KB = **120 MB/s**

Трафик в день:

≈ **10 TB**

Основная нагрузка будет приходиться на:

-   изображения блюд
-   фото ресторанов

Поэтому изображения должны обслуживаться через **CDN**.

# 4. Допущения, ограничения и открытые вопросы

## 4.1 Допущения (Assumptions)

Следующие предположения сделаны из-за отсутствия точных данных от
заказчика:

-   MAU системы достигает **20 млн пользователей**
-   DAU составляет **20% MAU**
-   5% пользователей делают заказ ежедневно
-   пик нагрузки в **5 раз выше среднего**
-   средний размер ответа API --- **50 KB**
-   изображения блюд хранятся в CDN
-   используется внешний платежный сервис

## 4.2 Ограничения

### Бюджет

Заказчик указал ограниченный бюджет, поэтому:

-   используется **облачная инфраструктура**
-   предпочтение **managed сервисам**
-   минимизация cross-region трафика

### Команда разработки

Предполагается небольшая команда:

-   4--6 backend разработчиков
-   2 frontend разработчика
-   1--2 DevOps

Это ограничивает сложность архитектуры.

### Внешние зависимости

Система зависит от:

-   платежных провайдеров
-   push notification сервисов
-   SMS провайдеров

## 4.4 Выявленные противоречия

### Противоречие 1

Требования заказчика:

-   **100% надежность платежей**
-   **очень быстрый UX**

Это противоречие, так как надежная обработка платежей требует
дополнительных проверок.

**Решение:**

-   использовать идемпотентные операции
-   асинхронную обработку платежей
-   event-driven архитектуру

------------------------------------------------------------------------

### Противоречие 2

Требования:

-   обновление статуса доставки **в реальном времени**
-   **ограниченный бюджет инфраструктуры**

**Решение:**

-   использовать WebSocket или SSE
-   отправлять обновления только при изменении статуса
-   использовать CDN и кэширование для снижения нагрузки
